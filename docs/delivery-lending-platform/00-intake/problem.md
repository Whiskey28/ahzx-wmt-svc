# 问题陈述（Intake）— 统一信贷业务技术平台

## 背景

公司计划在现有 **WMT Spring Boot 多模块工程** 上建设面向信贷/客群的业务能力，覆盖：

1. **微信小程序（C 端）**：客户注册登录、额度测算、贷款申请、信息授权与材料上传、准入结果查询、营销/申请进度跟踪、授信额度展示等。
2. **数字化管理平台（Web，内部）**：客户端与管理端能力、数据驾驶舱与可视化、业务流程自动化、智能获客与精准营销、风险全周期管理、监管报送等。
3. **贷后管理平台（业务域）**：贷后监测、风险预警、催收管理、贷后流程标准化与智能化运营。（实施上可作为管理平台的**子域**先交付，必要时再拆独立前端。）

目标形态：**两个前端 + 一个后端**；后端单仓聚合，服务全公司多人协作，**数据敏感**，需要 **RBAC + 数据权限分级**。

## 成功标准（初版）

- C 端与内部端共用 **同一套后端服务集群**（逻辑上可分包、分库分表，但对外一套 API 基座与统一安全体系）。
- 权限模型支持：组织/部门、角色、菜单、接口、**数据范围**（本人/本部门/自定义/全部等，与现有 `wmt-framework` 能力对齐）。
- 可剥离历史 **credit-report / report** 相关 Maven 依赖后，同一工程骨架仍可启动并承载新业务能力（模块化复用）。
- 后续可进入流水线阶段 **data_contracts → backend → frontend → qa → deploy**，并以 `check-gates.ps1` 作为质量门禁（需在仓库接入 `.delivery-pipeline` 模板后执行）。

## 非目标（第一期可先不做）

- 完整监管报送全量表结构落地（可先定义报送边界与对接方式）。
- 小程序与管理端 UI 像素级设计体系（先做信息架构 + OpenAPI 契约驱动）。
