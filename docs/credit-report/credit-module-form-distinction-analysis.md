# 征信模块表单区分字段设计方案分析报告

## 一、需求调研问题

在给出设计方案之前，需要明确以下关键问题：

### 1. 业务场景问题

**Q1：多表单的具体场景是什么？**
- 一个部门在一个周期内需要填写多个不同类型的表单吗？（例如：财务部在2025-01月需要填写"月度经营报表"和"月度财务分析表"）
- 还是同一个表单模板，但需要多个部门分别填写？（当前设计已支持）
- 或者是同一个部门，但不同角色的人填写不同的表单？（例如：财务部经理填写"汇总表"，财务部会计填写"明细表"）

**Q2：表单之间的关联关系是什么？**
- 多个表单的数据是否需要汇总计算？（例如：多个部门的表单数据汇总成总报表）
- 表单之间是否有依赖关系？（例如：必须先填写A表单，才能填写B表单）
- 不同表单的字段配置是否完全不同，还是部分重叠？

**Q3：权限控制需求是什么？**
- 不同角色的人是否应该看到不同的表单？
- 同一个用户是否可能同时拥有多个角色，需要填写多个表单？
- 表单的查看、编辑、提交权限是否需要按角色区分？

### 2. 数据组织问题

**Q4：报表生成逻辑是什么？**
- 当前计算规则使用 `#dept115`, `#dept116` 这样的表达式来汇总部门数据
- 如果改为按角色区分，计算规则如何表达？（例如：`#role1`, `#role2`？）
- 还是需要同时支持按部门和按角色两种维度？

**Q5：历史数据兼容性**
- 现有数据都是按部门组织的，如果改变区分字段，历史数据如何处理？
- 是否需要数据迁移方案？

### 3. 扩展性问题

**Q6：未来可能的扩展需求**
- 是否可能还需要按其他维度区分表单？（例如：按项目、按地区、按业务线）
- 是否需要支持更灵活的配置，让管理员可以自定义区分维度？

## 二、当前设计分析

### 2.1 当前架构

```
字段配置 (credit_field_config)
├── dept_id = 0 (通用字段)
└── dept_id = 115/116/117 (部门字段)

表单数据 (credit_form_data)
├── dept_id (部门ID)
├── report_period (报送周期)
├── report_type (报表类型：月报/季报)
└── form_data (JSONB格式的字段值)

计算规则 (credit_calculation_rule)
└── rule_expression: "#sum({#dept115[...], #dept116[...], #dept117[...]})"
```

### 2.2 当前设计的优势

✅ **组织架构清晰**：部门是稳定的组织单元，便于管理和权限控制  
✅ **数据汇总直观**：按部门汇总数据符合常见的报表需求  
✅ **权限控制简单**：用户属于某个部门，自然只能看到本部门的表单  
✅ **计算规则明确**：`#dept{id}` 的表达方式清晰易懂

### 2.3 当前设计的局限性

❌ **灵活性不足**：一个部门在一个周期内只能有一个表单  
❌ **扩展性受限**：如果需要支持多种表单类型，需要增加新的区分维度  
❌ **角色复用困难**：如果同一角色的人分布在多个部门，无法统一管理

## 三、方案对比分析

### 方案一：保持部门区分，增加表单类型字段

#### 设计思路
在现有基础上，增加 `form_type` 或 `form_template_id` 字段来区分不同类型的表单。

```sql
-- 字段配置表
ALTER TABLE credit_field_config 
ADD COLUMN form_type VARCHAR(50); -- 例如：'MONTHLY_REPORT', 'FINANCIAL_ANALYSIS'

-- 表单数据表
ALTER TABLE credit_form_data 
ADD COLUMN form_type VARCHAR(50); -- 与字段配置保持一致

-- 唯一约束
UNIQUE KEY uk_dept_period_type (dept_id, report_period, report_type, form_type);
```

#### 优势
✅ **改动最小**：在现有设计基础上扩展，兼容性好  
✅ **向后兼容**：可以为现有数据设置默认的 form_type  
✅ **计算规则清晰**：可以继续使用 `#dept{id}` 表达式，只需指定 form_type  
✅ **权限控制简单**：仍然基于部门，易于理解

#### 劣势
❌ **仍然受限于部门**：如果跨部门需要统一管理，仍然不够灵活  
❌ **角色支持不足**：无法直接支持按角色区分

#### 适用场景
- 一个部门需要填写多种类型的表单
- 表单类型相对固定，不会频繁变化
- 仍然以部门为主要管理维度

---

### 方案二：改为角色区分

#### 设计思路
将 `dept_id` 改为 `role_id`，按角色来区分字段配置和表单数据。

```sql
-- 字段配置表
ALTER TABLE credit_field_config 
CHANGE COLUMN dept_id role_id BIGINT; -- 0表示通用字段

-- 表单数据表
ALTER TABLE credit_form_data 
CHANGE COLUMN dept_id role_id BIGINT;

-- 计算规则表达式改为
"#sum({#role1[...], #role2[...], #role3[...]})"
```

#### 优势
✅ **灵活性高**：角色可以跨部门，更灵活  
✅ **权限控制精确**：可以精确控制哪些角色可以填写哪些表单  
✅ **支持多角色**：一个用户可以拥有多个角色，填写多个表单

#### 劣势
❌ **改动巨大**：需要修改所有相关代码和计算规则  
❌ **数据迁移复杂**：需要将部门映射到角色，历史数据处理困难  
❌ **组织关系弱化**：失去了部门的组织架构优势  
❌ **计算规则复杂**：如果角色和部门不对应，汇总逻辑可能混乱

#### 适用场景
- 表单填写与部门关系不大，主要按职责/角色区分
- 同一角色的人分布在多个部门，需要统一管理
- 权限控制需要精确到角色级别

---

### 方案三：部门 + 表单类型（推荐）

#### 设计思路
保留部门区分，同时增加表单类型字段，形成二维区分。

```sql
-- 字段配置表
ALTER TABLE credit_field_config 
ADD COLUMN form_type VARCHAR(50) DEFAULT 'DEFAULT'; -- 表单类型，默认值保持兼容

-- 表单数据表
ALTER TABLE credit_form_data 
ADD COLUMN form_type VARCHAR(50) DEFAULT 'DEFAULT';

-- 唯一约束
UNIQUE KEY uk_dept_period_type_form (dept_id, report_period, report_type, form_type);
```

#### 优势
✅ **兼顾灵活性和稳定性**：既保留了部门的组织优势，又支持多表单类型  
✅ **向后兼容**：通过默认值可以兼容现有数据  
✅ **扩展性好**：未来可以继续增加其他维度（如项目、地区等）  
✅ **计算规则清晰**：可以按部门+表单类型组合来引用数据

#### 劣势
❌ **复杂度增加**：需要同时管理部门和表单类型两个维度  
❌ **权限控制复杂**：需要同时考虑部门和表单类型的权限

#### 适用场景
- **最通用场景**：既需要部门管理，又需要支持多种表单类型
- 未来可能还需要扩展其他维度

---

### 方案四：引入表单模板/分类概念（最灵活）

#### 设计思路
引入独立的表单模板表，字段配置和表单数据都关联到模板。

```sql
-- 新增表单模板表
CREATE TABLE credit_form_template (
    id BIGINT PRIMARY KEY,
    template_code VARCHAR(50) UNIQUE, -- 模板编码
    template_name VARCHAR(100), -- 模板名称
    dept_id BIGINT, -- 关联部门（可选，NULL表示通用模板）
    role_id BIGINT, -- 关联角色（可选，NULL表示通用模板）
    report_type VARCHAR(20), -- 报表类型
    status INT, -- 状态
    ...
);

-- 字段配置表
ALTER TABLE credit_field_config 
ADD COLUMN template_id BIGINT; -- 关联表单模板

-- 表单数据表
ALTER TABLE credit_form_data 
ADD COLUMN template_id BIGINT; -- 关联表单模板
```

#### 优势
✅ **最灵活**：可以按部门、角色、项目等多种维度配置模板  
✅ **可扩展性强**：未来可以轻松增加新的区分维度  
✅ **配置集中**：表单模板统一管理，便于维护

#### 劣势
❌ **设计复杂**：需要新增表，改动较大  
❌ **学习成本高**：概念更抽象，需要更多培训

#### 适用场景
- 表单类型多样，区分维度复杂
- 需要高度灵活的配置能力
- 系统规模较大，需要长期维护

## 四、推荐方案

### 4.1 短期方案（推荐）：方案三 - 部门 + 表单类型

**理由：**
1. **改动最小**：只需增加一个字段，兼容现有数据
2. **满足需求**：可以支持一个部门在一个周期内填写多个表单
3. **保持优势**：仍然保留部门的组织架构优势
4. **易于理解**：概念简单，易于开发和维护

**实施步骤：**
1. 在 `credit_field_config` 和 `credit_form_data` 表中增加 `form_type` 字段
2. 为现有数据设置默认的 `form_type = 'DEFAULT'`
3. 修改唯一约束，包含 `form_type`
4. 修改计算规则表达式，支持按 `form_type` 过滤
5. 前端界面增加表单类型选择

### 4.2 长期方案（如果需求复杂）：方案四 - 表单模板

**理由：**
1. **最灵活**：可以应对未来各种扩展需求
2. **可维护性强**：配置集中管理，便于维护
3. **扩展性好**：可以轻松增加新的区分维度

## 六、实施建议

### 6.1 如果选择方案三（部门 + 表单类型）

**数据库变更：**
```sql
-- 1. 增加表单类型字段
ALTER TABLE credit_field_config 
ADD COLUMN form_type VARCHAR(50) DEFAULT 'DEFAULT';

ALTER TABLE credit_form_data 
ADD COLUMN form_type VARCHAR(50) DEFAULT 'DEFAULT';

-- 2. 修改唯一约束
ALTER TABLE credit_form_data 
DROP CONSTRAINT IF EXISTS uk_dept_period_type;

ALTER TABLE credit_form_data 
ADD CONSTRAINT uk_dept_period_type_form 
UNIQUE (dept_id, report_period, report_type, form_type);

-- 3. 为现有数据设置默认值（已通过DEFAULT完成）
```

**代码变更：**
- `CreditFieldConfigDO` 增加 `formType` 字段
- `CreditFormDataDO` 增加 `formType` 字段
- 修改查询逻辑，支持按 `formType` 过滤
- 修改计算规则表达式解析，支持按 `formType` 引用数据

### 6.2 如果选择方案四（表单模板）

**需要更详细的设计文档和迁移方案，建议分阶段实施：**
1. 第一阶段：新增表单模板表，但不强制使用
2. 第二阶段：逐步迁移现有配置到模板
3. 第三阶段：完全切换到模板模式

## 七、总结

**核心问题：** 您希望系统支持"一个部门在一个周期内填写多个不同类型的表单"，还是"跨部门按角色统一管理表单"？

**推荐路径：**
1. **如果需求明确且相对简单** → 选择**方案三（部门 + 表单类型）**
2. **如果需求复杂且可能频繁变化** → 选择**方案四（表单模板）**
3. **如果主要按角色管理** → 选择**方案二（角色区分）**，但需要充分评估迁移成本
