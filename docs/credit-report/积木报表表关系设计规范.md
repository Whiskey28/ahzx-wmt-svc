# 积木报表表关系设计规范

## 一、问题分析

在实际使用中，发现 `report_fill_enterprise_basic` 等 1:1 表的主键 `id` 与主记录 `report_fill_basic_info.id` 可能不相等，导致关联查询出现问题。

**原因分析：**
1. 积木报表在保存数据时，如果 1:1 表的主键 `id` 没有显式绑定到主记录的 `id`，会生成新的 UUID
2. 这导致 `enterprise_basic.id ≠ basic_info.id`，无法通过主键直接关联

## 二、推荐设计方案

### 方案一：1:1 表添加 `parent_id` 字段（推荐）

**优点：**
- 明确表达关联关系，语义清晰
- 不依赖主键相等，更灵活
- 便于查询和维护
- 与 1:N 表的设计保持一致

**表结构设计：**

```sql
-- 主记录表（关系表）
CREATE TABLE report_fill_basic_info (
  id          VARCHAR(32) NOT NULL PRIMARY KEY,  -- 主键，UUID
  period_id   VARCHAR(16) NOT NULL,
  role_id     VARCHAR(64) NOT NULL,
  report_id   VARCHAR(64) NOT NULL,
  -- ... 其他字段
);

-- 1:1 主数据块表（推荐方案：添加 parent_id）
CREATE TABLE report_fill_enterprise_basic (
  id            VARCHAR(32) NOT NULL PRIMARY KEY,  -- 独立主键，UUID
  parent_id     VARCHAR(32) NOT NULL,              -- 关联主记录（新增）
  enterprise_name VARCHAR(256),
  credit_code    VARCHAR(64),
  -- ... 其他业务字段
  -- 创建唯一索引，确保 1:1 关系
  CONSTRAINT uk_enterprise_basic_parent_id UNIQUE (parent_id)
);

-- 1:N 明细表（已有 record_id）
CREATE TABLE report_fill_product_stat (
  id           VARCHAR(32) NOT NULL PRIMARY KEY,  -- 独立主键，UUID
  record_id    VARCHAR(32) NOT NULL,              -- 关联主记录
  product_type VARCHAR(64) NOT NULL,
  -- ... 其他业务字段
  INDEX idx_product_stat_record_id (record_id)
);
```

**Java DO 设计：**

```java
// 1:1 表 DO
@TableName("report_fill_enterprise_basic")
public class ReportFillEnterpriseBasicDO extends BaseDO {
    @TableId(type = IdType.ASSIGN_UUID)
    private String id;              // 独立主键
    
    private String parentId;        // 关联主记录（新增）
    
    private String enterpriseName;
    // ... 其他字段
}

// 1:N 表 DO（保持不变）
@TableName("report_fill_product_stat")
public class ReportFillProductStatDO extends BaseDO {
    @TableId(type = IdType.ASSIGN_UUID)
    private String id;              // 独立主键
    
    private String recordId;         // 关联主记录
    
    // ... 其他字段
}
```

**积木报表配置：**

1. **关系表（report_fill_basic_info）数据集：**
   - 主键 `id`：使用 `${param.recordId}` 或自动生成 UUID
   - 其他字段正常绑定

2. **1:1 主数据块表（report_fill_enterprise_basic）数据集：**
   - 主键 `id`：自动生成 UUID（`IdType.ASSIGN_UUID`）
   - **`parent_id`**：绑定到关系表的 `id` 字段
     - 在积木报表中，将 `parent_id` 设置为隐藏字段
     - 默认值：`${report_fill_basic_info.id}` 或 `${param.recordId}`
   - 业务字段正常绑定

3. **1:N 明细表（report_fill_product_stat）数据集：**
   - 主键 `id`：自动生成 UUID
   - **`record_id`**：绑定到关系表的 `id` 字段
     - 默认值：`${report_fill_basic_info.id}` 或 `${param.recordId}`
   - 业务字段正常绑定

### 方案二：1:1 表主键等于主记录主键（当前方案，需严格配置）

**表结构设计：**

```sql
-- 1:1 主数据块表（主键 = 主记录主键）
CREATE TABLE report_fill_enterprise_basic (
  id            VARCHAR(32) NOT NULL PRIMARY KEY,  -- 必须等于 report_fill_basic_info.id
  enterprise_name VARCHAR(256),
  -- ... 其他业务字段
);
```

**积木报表配置要求：**

1. **关系表（report_fill_basic_info）数据集：**
   - 主键 `id`：使用 `${param.recordId}` 或生成 UUID，并保存到变量

2. **1:1 主数据块表（report_fill_enterprise_basic）数据集：**
   - **主键 `id`**：必须绑定到关系表的 `id` 字段
     - 在积木报表中，将 `id` 字段设置为隐藏字段
     - 默认值：`${report_fill_basic_info.id}`（必须使用关系表的 id）
   - 业务字段正常绑定

**注意事项：**
- 必须确保 1:1 表的主键 `id` 与关系表的 `id` 完全一致
- 如果积木报表自动生成了新的 UUID，会导致关联失败
- 需要在积木报表配置中严格绑定，不能依赖自动生成

## 三、两种方案对比

| 对比项 | 方案一（添加 parent_id） | 方案二（主键相等） |
|--------|-------------------------|-------------------|
| **语义清晰度** | ⭐⭐⭐⭐⭐ 明确表达父子关系 | ⭐⭐⭐ 依赖主键相等，不够直观 |
| **配置复杂度** | ⭐⭐⭐⭐ 需要配置 parent_id 绑定 | ⭐⭐⭐⭐⭐ 必须严格绑定主键 |
| **灵活性** | ⭐⭐⭐⭐⭐ 主键可独立生成 | ⭐⭐ 主键必须等于关系表主键 |
| **维护成本** | ⭐⭐⭐⭐ 需要维护唯一索引 | ⭐⭐⭐⭐⭐ 需要确保主键一致性 |
| **查询性能** | ⭐⭐⭐⭐ 通过 parent_id 索引查询 | ⭐⭐⭐⭐⭐ 通过主键查询，性能最好 |
| **与 1:N 表一致性** | ⭐⭐⭐⭐ 1:1用parent_id，1:N用record_id，语义区分清晰 | ⭐⭐ 设计不一致 |

## 四、推荐方案实施步骤

### 步骤 1：修改 SQL 脚本

为所有 1:1 表添加 `parent_id` 字段和唯一索引：

```sql
-- 修改 report_fill_enterprise_basic
ALTER TABLE report_fill_enterprise_basic 
  ADD COLUMN parent_id VARCHAR(32) NOT NULL;

-- 创建唯一索引，确保 1:1 关系
CREATE UNIQUE INDEX uk_enterprise_basic_parent_id 
  ON report_fill_enterprise_basic (parent_id);

-- 创建普通索引，便于查询
CREATE INDEX idx_enterprise_basic_parent_id 
  ON report_fill_enterprise_basic (parent_id);

-- 为现有数据填充 parent_id（如果 id 恰好等于主记录 id）
UPDATE report_fill_enterprise_basic e
SET parent_id = e.id
WHERE EXISTS (
  SELECT 1 FROM report_fill_basic_info b WHERE b.id = e.id
);

-- 对其他 1:1 表执行相同操作：
-- report_fill_biz_stat_finance
-- report_fill_biz_stat_hr
-- report_fill_biz_stat_credit_build
-- report_fill_complaint_security_stat
-- report_fill_info_collect_stat
```

### 步骤 2：更新 Java DO 类

为所有 1:1 表的 DO 类添加 `parentId` 字段：

### 步骤 3：更新 Mapper 查询方法

修改查询方法，使用 `parentId` 进行关联：


### 步骤 4：更新积木报表配置

在积木报表中，为所有 1:1 表的数据集添加 `parent_id` 字段绑定：

1. 打开数据集配置
2. 添加 `parent_id` 字段
3. 设置为隐藏字段
4. 默认值设置为：`${report_fill_basic_info.id}` 或 `${param.recordId}`

## 六、总结

**推荐使用方案一（添加 parent_id）：**

1. ✅ **语义清晰**：`parent_id` 明确表达父子关系，1:1 表用 `parent_id`，1:N 表用 `record_id`，语义区分清晰
2. ✅ **设计合理**：1:1 表用 `parent_id`（父子关系），1:N 表用 `record_id`（记录关联），命名更符合业务语义
3. ✅ **配置灵活**：不依赖主键相等，积木报表配置更简单
4. ✅ **易于维护**：通过唯一索引保证 1:1 关系，通过普通索引提升查询性能
5. ✅ **容错性强**：即使积木报表生成了新的 UUID，只要 `parent_id` 正确绑定即可

**实施建议：**
- 优先为所有 1:1 表添加 `parent_id` 字段
- 创建唯一索引确保 1:1 关系
- 创建普通索引提升查询性能
- 更新积木报表配置，绑定 `parent_id` 字段
- 更新 Java 代码，使用 `parentId` 进行关联查询
