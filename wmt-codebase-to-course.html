当模块没开：为什么你会看到“禁用”

把启动开关逻辑读懂，先定位问题再继续追代码

从一次请求开始

你调用了 Controller 里的 接口 时， 如果 wmt-module-bpm 没启用，系统不会继续执行业务，而是直接返回一个“已禁用”的响应。 把它当成“电源开关”：没通电就不让你运行。

CODE 
    @RequestMapping("/admin-api/bpm/**")
    public CommonResult<Boolean> bpm404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[工作流模块 wmt-module-bpm - 已禁用]bpm/ 开启]");
    }
PLAIN ENGLISH

这个默认 Controller 专门在模块没开启时兜底，避免你看到“404 找不到接口”。

当请求路径匹配 /admin-api/bpm/** 时，直接返回一个错误响应，提示你“请开启 bpm/”。

返回类型是 CommonResult<Boolean>，表示“统一响应格式 + 错误码 + 信息”。

把流程引擎“接管”：BPM 的 Flowable 配置

知道配置在哪儿，就能更快判断“为什么审批人/事件/执行方式不对”

配置就是“装配说明书”

你可以把 Flowable 看成“发动机”， 而 BpmFlowableConfiguration 就是“说明书”： 它告诉引擎哪些事件要监听、哪些“审核人候选人策略”要生效。

CODE 
@Bean(name = "applicationTaskExecutor")
@ConditionalOnMissingBean(name = "applicationTaskExecutor")
public AsyncListenableTaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(8);
    executor.setMaxPoolSize(8);
    executor.setQueueCapacity(100);
    executor.setThreadNamePrefix("flowable-task-Executor-");
    executor.setAwaitTerminationSeconds(30);
    executor.setWaitForTasksToCompleteOnShutdown(true);
    executor.setAllowCoreThreadTimeOut(true);
    executor.initialize();
    return executor;
}
PLAIN ENGLISH

这里声明了 Flowable 需要的 任务执行器，避免引擎启动时因为缺少 Bean 而报错。

设置核心/最大线程数、队列容量、线程命名前缀，让后续排查“慢/卡住”的问题更可定位。

最后把 executor 初始化好并返回，提供给 Spring 容器托管使用。

获得模型列表：Controller 拉起 Service，再由引擎查询

你会从“接口返回什么”快速推回“数据怎么来的”

让数据像寄信一样走“路径”

读代码时别急着全看完：先抓住 入口 调 Service 的那两三行， 它们往往就是数据流的“起点”。本项目里获取 BPM 模型列表时，先让 modelService 查出 Flowable 的 Model 列表，再进一步把表单/分类/部署信息拼成返回值。

CODE 
@GetMapping("/list")
@Operation(summary = "获得模型分页")
@Parameter(name = "name", description = "模型名称", example = "wmt")
public CommonResult<List<BpmModelRespVO>> getModelList(@RequestParam(value = "name", required = false) String name) {
    List<Model> list = modelService.getModelList(name);
    if (CollUtil.isEmpty(list)) {
        return success(Collections.emptyList());
    }

    // 获得 Form 表单
    Set<Long> formIds = convertSet(list, model -> {
        BpmModelMetaInfoVO metaInfo = BpmModelConvert.INSTANCE.parseMetaInfo(model);
        return metaInfo != null ? metaInfo.getFormId() : null;
    });
    Map<Long, BpmFormDO> formMap = formService.getFormMap(formIds);
PLAIN ENGLISH

这是一个 GET 接口：客户端要一个“模型列表”。

先调用 modelService.getModelList(name)：让 Service 去查询引擎里的模型。

如果没查到，就直接返回空列表，避免前端再猜“是不是接口报错”。

接着从模型元信息里解析出表单 ID，然后让 formService 批量取表单数据，拼装返回。

部署模型：先校验，再把模型变成流程定义

理解 deploy 这一步，就能快速定位“为什么部署失败/审批候选人不对”

草图进施工前要过关

把“部署（deploy）”想成：你要把一份流程设计稿提交到引擎里真正执行。 所以 deployModel 不是直接“保存就完事”，而是做一连串校验：模型存在、BPMN 图没问题、表单配置已完成、任务候选人策略已配置。

CODE 
@Override
@Transactional(rollbackFor = Exception.class) // 因为进行多个操作，所以开启事务
public void deployModel(Long userId, String id) {
    // 1.1 校验流程模型存在
    Model model = validateModelManager(id, userId);
    BpmModelMetaInfoVO metaInfo = BpmModelConvert.INSTANCE.parseMetaInfo(model);
    // 1.2 校验流程图
    byte[] bpmnBytes = getModelBpmnXML(model.getId());
    validateBpmnXml(bpmnBytes, metaInfo.getType());
    // 1.3 校验表单已配
    BpmFormDO form = validateFormConfig(metaInfo);
PLAIN ENGLISH

部署方法开启 事务，避免“部署一半失败导致状态脏”。

先用 validateModelManager 确认模型存在且当前用户有管理权限。

再取出 BPMN XML 字节并做校验，确保流程图能被引擎正确理解。

最后检查表单配置是否已经准备好，因为流程节点往往需要表单数据来运行。

把“为什么不生效”拆成可追踪的步骤

先把“业务表”当作文件夹

BPM 模块里不仅有引擎层（Flowable），还有业务层表结构（例如分类、表单等）。 Mapper 和 MyBatis-Plus 让“分类分页”和“落库校验”更清晰。 你可以把分类当成文件夹：先确保文件夹不重复，再决定翻哪一页。

CODE 
@Override
public Long createCategory(BpmCategorySaveReqVO createReqVO) {
    // 校验唯一
    validateCategoryNameUnique(createReqVO);
    validateCategoryCodeUnique(createReqVO);
    // 插入
    BpmCategoryDO category = BeanUtils.toBean(createReqVO, BpmCategoryDO.class);
    bpmCategoryMapper.insert(category);
    return category.getId();
}
PLAIN ENGLISH

创建分类时先做唯一性校验：名字不能重复、编码不能重复。

通过 insert 把业务对象写入表。

最后返回新分类的 id，供前端继续发起后续操作（例如创建模型时选择分类）。
